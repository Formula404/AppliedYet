# 投了吗｜技术文档

> 本地优先的个人求职全流程桌面应用技术设计。

---

## 1. 文档信息

- 产品名称：投了吗
- 文档类型：技术设计文档
- 目标平台：Windows 10 / Windows 11
- 文档版本：1.0
- 文档日期：2026-07-13
- 核心技术：Tauri、React、TypeScript、Rust、Python、SQLite、FFmpeg
- 核心原则：本地优先、模块解耦、任务可恢复、自动化可撤销、凭据不落明文

---

## 2. 技术目标

系统需要同时满足以下能力：

1. 提供稳定的 Windows 桌面应用；
2. 本地保存投递、邮件索引、简历、面试和 Offer 数据；
3. 支持 OAuth 和 IMAP 读取招聘邮件；
4. 识别邮件类型并自动匹配岗位；
5. 支持 AI 生成岗位准备材料和模拟面试；
6. 支持音频、视频预处理和语音识别；
7. 支持真实面试问答切分与复盘；
8. 支持后台任务、失败重试和中断恢复；
9. 安全保存 API Key、邮箱授权码和 Token；
10. 支持完整备份、恢复和数据导出；
11. 支持打包成独立安装包分发给其他用户；
12. 用户无需自行安装 Python、Node.js 或 FFmpeg。

---

## 3. 非功能要求

### 3.1 性能

- 应用冷启动目标：5 秒内；
- 普通页面切换：500 毫秒内完成首屏响应；
- 本地搜索：1 秒内返回常见规模结果；
- 邮件增量扫描不阻塞界面；
- 音频处理、AI 和 ASR 均在后台运行；
- 10,000 条邮件索引和 5,000 条问题记录下保持可用。

### 3.2 稳定性

- 后台任务失败不得破坏已有业务数据；
- 关键写入使用事务；
- 应用异常退出后可恢复未完成任务；
- 邮件扫描必须幂等；
- 数据库迁移必须可回滚或备份后执行。

### 3.3 安全

- API Key、邮箱密码、授权码和 Token 不写入普通配置文件；
- SQLite 中不保存明文凭据；
- 日志必须脱敏；
- 外部服务调用必须记录数据类型和用途；
- 邮件权限遵循最小权限。

### 3.4 可维护性

- 前端、桌面核心、AI 工作流、邮箱、ASR 和存储层解耦；
- 第三方 AI、ASR 和邮箱服务采用适配器接口；
- 数据库通过版本化迁移管理；
- 关键对象使用明确的类型定义和 JSON Schema。

## 6. 进程模型

建议采用以下进程结构：

```text
投了吗.exe
├── Tauri 主进程
├── WebView 前端
├── Python Sidecar
└── FFmpeg 子进程（按需启动）
```

### 6.1 Tauri 主进程

负责：

- 应用生命周期；
- 单实例；
- 数据库；
- 文件系统；
- 凭据；
- 托盘；
- 通知；
- 启动 Python；
- 关闭时回收进程。

### 6.2 Python Sidecar

建议以本地 HTTP 服务方式运行：

- 仅监听 `127.0.0.1`；
- 启动时动态分配端口；
- Tauri 生成一次性鉴权 Token；
- 每个请求携带本地 Token；
- 禁止局域网访问；
- 禁止开放 Swagger 到生产环境；
- 支持健康检查；
- 支持优雅关闭。

### 6.3 FFmpeg

- 由 Rust 或 Python 调用；
- 每个任务独立子进程；
- 捕获标准输出和错误；
- 支持取消；
- 限制并发数量；
- 输出文件写入任务临时目录。

## 8. 数据目录

建议使用 Windows 标准应用数据目录：

```text
%APPDATA%\TouLeMa\
├── app.db
├── backups/
├── resumes/
├── source_materials/
├── emails/
├── interviews/
│   ├── original/
│   ├── processed/
│   ├── transcripts/
│   └── exports/
├── temp/
├── logs/
└── cache/
```

原则：

- 数据库和附件分离；
- 数据库存相对路径；
- 文件使用 UUID 命名；
- 原始文件名保存在元数据中；
- 临时目录可安全清理；
- 删除业务记录时根据引用计数决定是否删除文件。

## 12.7 岗位匹配评分

推荐评分因子：

| 因子 | 权重建议 |
|---|---:|
| 岗位编号完全一致 | 40 |
| 公司名称完全一致 | 25 |
| 岗位名称相似 | 20 |
| 发件域名匹配公司 | 15 |
| 邮件线程历史匹配 | 20 |
| 投递时间接近 | 10 |
| 工作地点一致 | 5 |
| 当前阶段合理 | 10 |
| 同公司多岗位歧义 | -20 |
| 邮件缺少岗位信息 | -15 |

最终归一化为 0–1。

默认阈值：

- `>= 0.85`：自动；
- `0.55–0.85`：待确认；
- `< 0.55`：不匹配。

阈值需要支持用户设置。

## 12.8 幂等与撤销

- 每封邮件只生成一次主要匹配决策；
- 重扫不得重复创建事件；
- 撤销时不删除邮件；
- 撤销记录写入新事件；
- 用户确认后的人工决策优先于后续自动判断。

## 14.4 分片策略

- 优先按静音分割；
- 片段间保留重叠；
- 合并时去重；
- 保存片段映射；

## 28. 安全威胁与防护

### 28.1 恶意邮件内容

风险：

- Prompt Injection；
- HTML 脚本；
- 超长内容；
- 恶意附件。

防护：

- 邮件正文视为不可信输入；
- 去除脚本；
- 不执行 HTML；
- AI Prompt 明确忽略邮件中的指令；
- 限制长度；
- 默认不自动下载附件；
- 附件解析使用白名单。

### 28.2 本地服务暴露

防护：

- 仅监听 localhost；
- 随机端口；
- 随机 Token；
- 请求体限制；
- 文件路径白名单；
- 不开放调试文档。

### 28.3 路径遍历

防护：

- 所有文件路径规范化；
- 禁止 `..`；
- 限制在应用数据目录或用户明确授权文件；
- 导出使用安全文件名。

### 28.4 数据库注入

防护：

- 所有 SQL 参数化；
- 禁止拼接用户输入；
- FTS 查询转义；
- 迁移脚本只从签名资源读取。

### 28.5 凭据泄露

防护：

- 使用系统凭据；
- 日志脱敏；
- UI 只显示掩码；
- 剪贴板复制后可自动清空；
- 错误上报不包含凭据。

## 30. 关键技术决策

### 30.1 为什么采用本地优先

- 简历、邮箱和录音高度敏感；
- 用户自行配置第三方服务；
- 无需维护中心化后端；
- 便于个人使用和离线查看；
- 安装包可直接分发。

### 30.2 为什么保留 Python Sidecar

- 文本和 AI 工作流开发效率高；
- 文档解析生态成熟；
- ASR 和 NLP 集成方便；
- 便于独立测试和维护。

### 30.3 为什么业务数据库由 Rust 控制

- 避免两个进程同时直接写数据库；
- 统一事务；
- 降低锁冲突；
- 便于权限控制；
- Python 通过 API 返回结果，Rust 负责最终落库。

### 30.4 为什么邮件采用规则优先

- 降低调用成本；
- 降低隐私外发；
- 常见招聘邮件规则明确；
- AI 只处理不确定情况；
- 更容易解释和撤销。

## 33. 总结

“投了吗”的技术核心不是单独实现一个投递表、邮箱客户端或 AI 对话框，而是建立一个稳定的本地数据底座，把以下对象可靠关联起来：

```text
公司
  ↓
岗位
  ↓
投递
  ├── 简历
  ├── 邮件
  ├── 流程事件
  ├── 任务
  ├── 面经
  ├── 模拟面试
  ├── 真实面试
  ├── 问题与故事
  └── Offer
```

所有自动化能力都必须具备三个特征：

1. 可解释；
2. 可校正；
3. 可撤销。

只有这样，“投了吗”才能从一个功能集合成长为可靠、长期可用的个人求职工作台。
