# 投了吗（Applied Yet?）

> 从投递到入职，把每一步都记清楚。

“投了吗”是一款面向 Windows 的本地优先个人求职工作台。它以一次具体的岗位投递为中心，把公司、岗位、简历、招聘邮件、面试准备、模拟面试、真实面试复盘与 Offer 关联在同一条可追溯的求职流程中。

当前仓库已完成产品基础骨架和全部一级页面，可作为后续接入 SQLite、邮箱、AI、ASR 与本地文件系统的前端基础。

## 当前已实现

- 桌面工作台：固定侧边栏、顶部搜索、快捷操作、通知与邮件同步反馈；
- 首页：月历、KPI、投递漏斗、今日待办、最近邮件、本周进展与重点岗位；
- 我的投递：岗位看板/列表切换、搜索筛选、新增投递表单；
- 招聘邮件：邮件列表、识别结果、匹配依据、置信度与确认更新流程；
- 面试准备：岗位理解、简历匹配、准备清单与预测问题；
- 模拟面试：岗位、类型、难度、题量、语音/文字方式等配置；
- 面试复盘：本地文件导入入口、待复盘记录与持续改善项；
- 个人题库：问题库、故事库与薄弱项入口；
- Offer：多 Offer 维度对比和用户自定义权重的界面基础；
- 数据分析：投递转化、岗位分布和可执行建议；
- 设置：本地数据位置、自动备份、恢复、导出与危险操作；
- 全局交互：`Ctrl + K` 搜索、`Ctrl + N` 新增投递、减少动态效果适配；
- Tauri 2 桌面入口：Windows 窗口尺寸、最小尺寸和基础权限配置。

当前页面使用演示数据，邮件同步、保存投递等操作用于展示完整交互状态，尚未写入真实数据库。

## 技术栈

- 桌面框架：Tauri 2 / Rust
- 前端：React 18、TypeScript、Vite
- 路由：React Router
- 图标：Lucide React
- 样式：原生 CSS Design Tokens，未依赖重量级 UI 框架
- 规划中的本地能力：SQLite + FTS5、Python Sidecar、FFmpeg

项目结构遵循产品技术文档的分层建议：

```text
.
```

## 本地运行

环境要求：

- Node.js 20 或更高版本；
- npm 10 或更高版本；
- 如需运行桌面壳：Rust stable 与 Tauri 2 的 Windows 系统依赖。

安装依赖：

```bash
cd apps/desktop
npm install
```

返回仓库根目录启动浏览器开发模式：

```bash
npm run dev
```

默认访问地址为 `http://localhost:1420`。

运行 Tauri 桌面模式：

```bash
npm run tauri dev
```

执行类型检查与生产构建：

```bash
npm run typecheck
npm run build
```

## 设计原则

实现遵循 `docs/投了吗_UI设计规范.md`：

- 日历是首页主要工作区，而不是装饰；
- 蓝色用于主要操作，绿/橙/紫/青/红严格表达业务状态；
- 首页保持高信息密度，但通过卡片、间距和字号层级控制阅读节奏；
- 自动匹配展示来源、依据、置信度，并保留用户确认入口；
- AI 不占据首页中心，也不替用户做 Offer 决策；
- 最小窗口尺寸为 1024 × 720，并适配 1440 × 900 的主要设计基准。

## 后续开发建议

建议继续按照技术文档的依赖顺序推进：

1. 建立 SQLite 迁移、公司/岗位/投递/任务仓储层；
2. 将演示数据替换为 Tauri Command 与 TanStack Query 数据流；
3. 接入 Windows Credential Manager，确保凭据不落明文；
4. 实现 OAuth/IMAP 增量同步、规则优先的邮件分类与可撤销更新；
5. 增加 Python Sidecar，接入可配置的 AI、文档解析与 ASR Provider；
6. 完成备份恢复、后台任务、错误重试、日志脱敏与安装包验证。

## 隐私说明

项目坚持本地优先：业务数据、简历、邮件索引和面试文件默认保存在用户设备上。仅在用户主动启用 AI、语音识别或邮箱能力时连接第三方服务；正式实现中应在每次外部调用前明确数据类型、用途与发送范围。
